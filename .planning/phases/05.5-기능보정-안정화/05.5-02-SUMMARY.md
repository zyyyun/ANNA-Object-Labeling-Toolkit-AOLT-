# Phase 05.5 Plan 02: RELI-06 Gap Closure Summary

Phase 5.5-01의 타임라인 단일 가드로는 막지 못했던 cold OpenCV 디코더 구간(~2-3초) blocking `videoCapture.Set(PosFrames, N)` 키프레임 재검색 폭주를 `_isVideoReady` 통합 게이트와 first-paint handshake로 근본 차단.

## Root Cause Recap

5.5-01 이후 UAT Test 4 재검증 시 "로드 직후 ~1fps 지연" 재현. 진단(RELI-06-diagnosis.md) 결과:

- 타임라인 가드 이외 7개 LoadFrame 진입 경로(타이머 33ms 주기, Rewind/Forward, Shift+화살표, 화살표, Waypoint 리스트 클릭 3종, TryNavigateToMarker)가 여전히 cold decoder 창에서 keyframe seek를 유발.
- `VideoService.LoadVideoAsync`의 `LoadFrame(0)` (line 145)과 `MainForm.LoadVideoWithSubtitle`의 `_videoService.LoadFrame(0)` (line 302) 이 **동일 seek를 2회** 실행.
- 자동재생(`btnPlay_Click`)이 LoadLabelingData 반환 직후 호출되어 첫 프레임 페인트 전에 timerPlayback이 기동 → cold-seek 누적.

## First-Paint Handshake Flow

1. `LoadVideoWithSubtitle` 시작 → `_isVideoReady = false`, `_pendingAutoPlay = false`
2. `VideoService.LoadVideoAsync(filePath)` 완료 (비디오 캡처 오픈, 중복 LoadFrame(0) 제거됨)
3. `var bitmap = _videoService.LoadFrame(0);` — **직접 VideoService 호출** (MainForm.LoadFrame 래퍼의 ready-gate 우회, 첫 비트맵 확보)
4. `pictureBoxVideo.Image = bitmap;`
5. `await LoadLabelingData(filePath);` — cold-decoder 기간, 이 동안의 모든 LoadFrame 진입은 guards에 의해 차단
6. `_pendingAutoPlay = true;`
7. WinForms가 pictureBoxVideo를 실제로 페인트 → `pictureBoxVideo_Paint` 핸들러 말미:
   - `_isVideoReady = true` 로 1회 전이
   - `_pendingAutoPlay`가 true면 소비 후 `btnPlay_Click(null, EventArgs.Empty)` — 정상 재생 시작
8. 이후 모든 LoadFrame 경로는 가드 통과 → cold-decoder 지연 없음
